# 2026-04 Decisions

## 2026-04-07

### Decision: Working product name is `DocMesh`

Reason:

- clear enough to convey document collaboration
- broad enough to avoid overfitting to wiki-only positioning
- fits the idea of a networked document graph for agents

### Decision: Repo follows LLM Wiki-style documentation from day one

Reason:

- important repo context should survive chat sessions
- architecture and product direction will evolve quickly
- durable decisions should be written into repo docs as the source of truth

### Decision: `AGENTS.md` is a table of contents, not a monolithic manual

Reason:

- keeps the injected context small
- aligns with progressive disclosure for agents
- reduces rot and stale rule accumulation

### Decision: Use a structured docs set instead of a single monolithic design doc

Reason:

- easier for future agents to update the right place
- separates stable knowledge from high-churn planning artifacts
- supports gradual migration into the future hosted knowledge service

### Decision: Initial implementation stack is Go + Gin + Cobra + Zap + PostgreSQL

Reason:

- matches the desired engineering direction
- keeps the first version conventional and easy to extend
- avoids framework novelty during the critical model-design phase

### Decision: `cmd/cli` remains a thin HTTP wrapper

Reason:

- keeps service logic centralized in the server
- avoids duplicating business logic in CLI code
- makes future API-first integrations easier

### Decision: Use direct `pgx` SQL for v0 instead of adding ORM or codegen first

Reason:

- keeps the initial persistence layer boring and inspectable
- avoids premature schema abstraction while the data model is still moving
- leaves room to adopt `sqlc` later if query volume grows
